# 技术演讲文章写作

将演讲重写为一篇有证据链的中文技术文章，而不是按时间轴翻译字幕。读者应能看懂问题、验证推理、知道结论边界，并能在需要时回到原始幻灯片。

## 写作工作流

1. 先读用户的拆分建议、文章草稿、双语字幕和 PDF。若主题可拆为数据结构、并发通信、性能测量等独立主线，先确定当前文章只解决哪一个工程问题。
2. 以英文原文、PDF 代码和图表作为技术事实来源；中文译文只用于语境和自然表达。译文冲突、数字不明或结论不确定时，使用条件化表述，不得补造事实。
3. 组织为“看似正确的起点 → 测量发现 → 机制解释 → 改动 → 证据 → 适用边界”的叙事链。每个优化都回答：原方案为何合理、为何在此工作负载变慢、改动转移了什么成本、图表证明了什么、何时不应照搬。
4. 用口语化、工程师分享式中文。保留悬念和反直觉转折；避免逐段复述演讲、教材腔、空泛形容词和“某技术永远更快”的结论。
5. 交付 Markdown 和相对路径素材。C++ 片段只在解释必要时保留；遵守当前工作区的代码规范和格式化要求，不在文章中暴露凭据或内部连接信息。

## 必须保留的技术深度

- 从业务消息和真实数据规模倒推数据结构，解释 ID 索引、插入/修改/删除语义、内存局部性、迭代器失效和尾延迟。
- 性能章节按证据链展开：隔离初始化；先用 Top-Down（Retiring、Bad Speculation、Frontend Bound、Backend Bound）分类；再用 `perf record`、`perf report`、`perf annotate` 定位；最后读反汇编和硬件计数器。说明采样与单次延迟测量的边界，并保留优化构建的调试信息。
- 对编译结果给出原始 PDF 的相关图片和中文读图说明。区分语言层面的 `inline` 与编译器内联展开；同时解释分支布局、`[[likely]]` / `[[unlikely]]`、冷错误路径、I-cache，以及 lambda/functor 保留具体类型与 `std::function` 类型擦除的取舍。
- 在文末汇总演讲中的工程原则；将 Q&A 作为“适用边界/常见误解”写入，清楚标注它是对特定提问的回答。跨主题的 Q&A 可简述并留给系列后文，不应抢占当前主线。

## PDF 配图规则

- 只选能推进论点的原图：数据结构、延迟分布、热点数据、perf/汇编、硬件计数器、最终对比。不要堆砌整页幻灯片或装饰图片。
- 每张图紧邻解释它的段落；提供准确 alt 文本、中文图注、来源与演讲幻灯片页码。图注必须说明读者应看到的数值或关系。
- 将图片保存到文章旁的 `assets/` 子目录，并使用相对路径。交付前逐一验证路径存在、图号连续、图表前后结论一致。

## 发布到知乎

知乎导入器不能读取本机的 `assets/...` 相对路径。导入前，将图片改为可公开访问的 HTTPS URL，或先导入正文后在知乎编辑器中逐张上传。不要把 Windows 绝对路径、工作区路径或内部资源地址写进发布版 Markdown。

## 交付前检查

- 主线单一且完整；引言给出业务动机，结尾给出可复用但有条件的检查清单。
- 所有数字、代码、汇编解释和性能归因均能追溯到原始材料；实验结论明确标注机器、数据集、编译器或工作负载依赖性。
- 确认 Markdown 标题层级、代码围栏、链接、图片引用和图号有效；没有待办占位、重复段落或未解释的术语。
- 在声称完成前，重新检查目标文件及全部素材引用，并简要报告验证结果。
